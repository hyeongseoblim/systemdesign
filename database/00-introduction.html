2. 인덱스 & B+Tree

인덱스(Index)는 특정 컬럼에 대한 정렬된 별도 자료구조로, 풀 스캔(O(n)) 대신 O(log n) 탐색을 가능하게 한다. MySQL InnoDB와 PostgreSQL은 모두 B+Tree를 기본 인덱스 구조로 사용한다.

B+Tree 구조

[20,30][50,60][100,110][150,160]   ← 리프 노드: 실제 데이터(또는 row pointer)

면접 단골 질문
"인덱스가 있는데 왜 풀 스캔이 나오나요?" — 옵티마이저는 선택도(Selectivity)가 낮으면(예: 전체 데이터의 20% 이상 반환) 인덱스보다 풀 스캔이 더 빠르다고 판단한다. 또한 함수 적용(WHERE YEAR(created_at)=2024), 묵시적 타입 변환, Null 값 비교도 인덱스를 타지 못한다.
인덱스 함정
인덱스는 쓰기 비용을 증가시킨다(INSERT/UPDATE/DELETE 시 인덱스도 갱신). 불필요한 인덱스는 DML 성능을 저하시키고 디스크를 낭비한다. 실행 계획(EXPLAIN ANALYZE)으로 반드시 검증한다.

3. 트랜잭션 & 격리 수준

트랜잭션(Transaction)은 ACID 속성을 보장하는 작업 단위다.

  • Atomicity(원자성): 전부 성공 or 전부 실패
  • Consistency(일관성): 트랜잭션 전후 DB 무결성 제약 유지
  • Isolation(격리성): 동시 실행 트랜잭션이 서로 간섭하지 않음
  • Durability(영속성): 커밋된 데이터는 시스템 장애 후에도 유지

격리 수준(Isolation Level)별 발생 현상

T2->>DB: ROLLBACK Note over T1: T1이 존재하지 않았던 값을 읽음 Note over T1,T2: READ COMMITTED — Non-repeatable Read 발생 T1->>DB: SELECT balance → 300 T2->>DB: UPDATE balance = 500 / COMMIT T1->>DB: SELECT balance → 500 (같은 트랜잭션 내 다른 값!) InnoDB MVCC
MySQL InnoDB의 REPEATABLE READ는 MVCC(Multi-Version Concurrency Control, 다중 버전 동시성 제어)를 통해 Phantom Read를 실질적으로 방지한다. 각 행에 숨겨진 버전 번호(DB_TRX_ID)를 기록하고, 트랜잭션 시작 시점의 스냅샷을 읽어 일관된 뷰를 제공한다.

단일 DB로 처리 불가한 대용량 데이터를 수평 분할하여 여러 DB 서버에 분산한다.

Consistent Hashing링(Ring) 구조. 노드 추가/제거 시 최소 이동리샤딩 최소화핫스팟 주의 (Virtual Node로 완화)

6. 쿼리 튜닝 기초

성능 문제의 80%는 느린 쿼리에서 발생한다. EXPLAIN / EXPLAIN ANALYZE 출력을 읽는 능력이 핵심이다.

EXPLAIN 핵심 컬럼 (MySQL)

컬럼의미주의 신호
type접근 방식. ALL → index → range → ref → eq_ref → const 순으로 좋음ALL = 풀 스캔
key실제 사용된 인덱스NULL = 인덱스 미사용
rows옵티마이저 예상 스캔 행 수큰 수 = 비효율
Extra부가 정보Using filesort, Using temporary 주의
filteredWHERE 조건으로 필터링되는 비율 (%)낮으면 인덱스 효과 ↓

N+1 문제

ORM(Object-Relational Mapping) 사용 시 가장 흔한 성능 함정이다.

# BAD: N+1 쿼리 (주문 100개 → 쿼리 101번)
orders = Order.findAll()          # 쿼리 1번
for order in orders:
    items = order.items           # 각 주문마다 쿼리 1번 × 100 = 100번

# GOOD: Fetch Join / Eager Loading (쿼리 1번)
orders = Order.findAll(join: :items)
쿼리 튜닝 체크리스트
  1. EXPLAIN으로 실행 계획 확인 (type=ALL 제거)
  2. 인덱스 컬럼에 함수/형변환 사용 금지
  3. SELECT * 대신 필요한 컬럼만 조회
  4. 페이지네이션: OFFSET 대신 Cursor 방식 사용 (대용량 테이블)
  5. ORM N+1 → Fetch Join 또는 별도 쿼리로 해결
  6. Slow Query Log 분석 (long_query_time=1s 설정)

"MySQL과 Redis를 같이 쓰는 이유는?" — Redis는 인메모리(In-memory) DB이므로 MySQL의 느린 복잡한 쿼리를 캐싱해 응답 시간을 ms 단위로 줄인다. Cache-Aside(캐시 어사이드) 패턴: 캐시 미스 시 DB에서 읽어 Redis에 저장, TTL로 만료 관리. Cache Stampede(캐시 스탬피드)와 Cache Invalidation(캐시 무효화) 전략도 함께 답해야 한다.
