2. 🧠 OMS — Order Management System (주문관리)

핵심 책임
주문이라는 비즈니스 객체의 라이프사이클 전체를 관리한다.

주문 상태 머신 (State Machine)

stateDiagram-v2 [*] --> PLACED : OrderPlaced PLACED --> PAID : PaymentCaptured PLACED --> CANCELED : CancelByCustomer PAID --> ALLOCATED : AllocationDecided PAID --> CANCELED : CancelBeforeAllocation ALLOCATED --> FULFILLING : PickingStarted ALLOCATED --> CANCELED : CancelBeforeFulfill FULFILLING --> SHIPPED : OutboundShipped SHIPPED --> DELIVERED : DeliveryConfirmed SHIPPED --> DELIVERY_FAILED : DeliveryFailed DELIVERY_FAILED --> SHIPPED : ReAttempt DELIVERY_FAILED --> RETURNED : ReturnToSender DELIVERED --> CLOSED : AfterReturnWindow DELIVERED --> RETURNED : ReturnRequested RETURNED --> REFUNDED : RefundProcessed CANCELED --> [*] CLOSED --> [*] REFUNDED --> [*]
주문 상태 전이 다이어그램 — Boolean 플래그 대신 이 상태 머신으로 강제해야 함

왜 WMS / TMS 에서 분리되는가

  • 주문은 여러 채널(자사 앱·오픈마켓·B2B)에서 들어오고, 여러 창고·파트너에 할당된다. 멀티 소스·멀티 싱크를 흡수하는 허브 역할이 필요.
  • 결제·쿠폰·정산·고객 CS 가 모두 주문에 엮여 있어, WMS/TMS 에 섞으면 Bounded Context(경계 있는 컨텍스트)가 무너진다.

내부 주요 기능

  • 주문 수집 & 검증: 재고·결제·주소 정규화·Fraud(사기) 검증
  • Allocation(할당): 어느 창고에서 어느 SKU 를 보낼지 결정. 단일창고 / 분할출고(Split shipment) / 드랍쉬핑(Dropshipping) 판단
  • 결제/정산 연동: Auth(인증) → Capture(매입) → Refund(환불). 부분 취소·부분 환불 처리
⚠️ 실무 함정
주문 상태를 Boolean 플래그 여러 개로 관리 → 조합 폭발. Enum + 상태 머신으로 강제해야 함.
분할 출고 시 "주문 1건 = 배송 1건" 가정이 깨짐. Order ↔ Shipment가 1:N.
Cut-off time(마감시각) 전후 주문 몰림 → 동시성 + 트랜잭션 경쟁 🔥 Deep-dive

3. 🏭 WMS — Warehouse Management System (창고관리)

핵심 책임
실물 재고의 입고부터 출고까지, 창고 내부 작업 전체를 관리한다.

재고 상태 분류

flowchart LR OH["On-hand\n실물 재고\n창고에 실제 존재"] RS["Reserved\n예약 재고\n주문에 할당됨"] AV["Available\n가용 재고\n= On-hand − Reserved"] IT["InTransit\n이동 중\n창고 간 이동"] DM["Damaged\n손상/폐기"] SP["Shipped\n출고 완료"] OH -->|"주문 Allocate"| RS OH --> AV OH --> DM OH --> IT RS -->|"Commit (출고 확정)"| SP RS -->|"Expire / Cancel"| AV style OH fill:#fef3c7,stroke:#f59e0b style RS fill:#fce7f3,stroke:#ec4899 style AV fill:#dcfce7,stroke:#22c55e style IT fill:#dbeafe,stroke:#3b82f6 style DM fill:#fee2e2,stroke:#ef4444 style SP fill:#e0e7ff,stroke:#6366f1
재고 상태 분류 — On-hand ≠ Available 임을 항상 구분할 것

Reserve → Commit → Ship 3단계

sequenceDiagram participant OMS as OMS participant Inv as WMS Inventory participant Pick as Picker participant Ship as Shipment OMS->>Inv: ReserveRequest(sku, qty, orderId, TTL) Inv->>Inv: Available -= qty / Reserved += qty Inv-->>OMS: Reserved(reservationId) Note over Inv: 만료 TTL 내에 Commit 없으면 자동 롤백 OMS->>Inv: CommitRequest(reservationId) Inv->>Pick: CreatePickList Pick->>Inv: PickCompleted Inv->>Inv: On-hand -= qty / Reserved -= qty Inv-->>OMS: Committed Inv->>Ship: ShipmentReady(packageId) Ship-->>OMS: OutboundShipped
재고 Reserve → Commit → Ship — 이 흐름을 모르면 면접에서 탈락한다
🎯 면접 포인트
"재고 차감을 어떻게 설계하시겠어요?" → "그냥 -1 하면 됩니다"는 즉시 탈락.
Reserve → Commit → Ship 3단계 + 만료 처리 + 원자적 조건부 UPDATE(또는 낙관적 락)를 이야기해야 함. 🔥 Deep-dive

피킹 전략 비교

전략방식적합한 상황
Batch picking여러 주문을 한 번에 묶어 피킹SKU 집중, 동선 반복
Zone picking창고 구역별 담당자 분리창고가 크고 구역 특성이 뚜렷할 때
Wave picking특정 시간대에 집중 처리Cut-off 기반 새벽배송 (컬리 샛별)
Cluster picking카트 여러 칸에 여러 주문 동시 처리중간 규모, 균형 효율
⚠️ 실무 함정
Oversell(초과판매): 예약 만료를 안 걸어서 주문은 받았는데 실물이 없음. 🔥 Deep-dive
창고 간 이동(Transfer) 중 "출발지에서 빠졌는데 도착지에 아직 안 찍힌" 상태 → InTransit 명시 필요.

4. 🚛 TMS — Transportation Management System (운송관리)

핵심 책임
화물을 언제·어떻게·얼마에 목적지까지 보낼지 결정하고 집행한다.

허브앤스포크 (Hub-and-spoke) 네트워크

flowchart LR P1[판매자 A] --> OH P2[판매자 B] --> OH P3[판매자 C] --> OH OH["🏢 서울 메가허브\n발송 Hub"] OH -->|"간선 트럭\n야간 운송"| DH DH["🏢 부산 메가허브\n도착 Hub"] DH --> C1["📍 캠프 A\n해운대"] DH --> C2["📍 캠프 B\n동래"] C1 --> D1((기사 1)) C1 --> D2((기사 2)) C2 --> D3((기사 3)) D1 --> G1([고객 ①]) D2 --> G2([고객 ②]) D3 --> G3([고객 ③]) style OH fill:#ede9fe,stroke:#8b5cf6 style DH fill:#ede9fe,stroke:#8b5cf6 style C1 fill:#fef3c7,stroke:#f59e0b style C2 fill:#fef3c7,stroke:#f59e0b
허브앤스포크 네트워크 — CJ대한통운·FedEx 표준 모델. 쿠팡은 이 모델의 하이브리드.

⚠️ 실무 함정
기사 앱 오프라인 동기화: 지하·산간에서 스캔 누락 후 재접속 시 이벤트 재생 필요.

5. 📦 Last-mile (라스트마일)

비용과 만족도의 결정적 구간
전체 물류 비용의 40–60%를 차지하며, 고객이 직접 경험하는 유일한 구간

주요 시나리오 처리

시나리오처리 방식시스템 대응
정상 배송기사가 직접 전달 + POD 촬영DELIVERED 이벤트 → OMS 상태 동기화
부재1회 재방문 / 문앞 배송 / 보관함DELIVERY_FAILED → ReAttempt 일정 등록
수령 거부반송 (Return to Sender) 처리RETURNED → OMS 반품 흐름 트리거
오배송회수 + 재배송 or 보상별도 ClaimEvent, CS 연동 필요
🎯 면접 포인트 (최상위 단골)
수천만 건/일 규모의 TrackingEvent Fan-out 설계: Kafka + CDC(Change Data Capture) + 최종 일관성(Eventual Consistency) 은 대형 이커머스 시스템 디자인 면접의 최상위 단골 주제. 🔥 Deep-dive

6. 쿠팡 로켓배송 — 수직 통합 사례

레이어쿠팡에서의 구현핵심 특징
OMS 쿠팡 앱 주문 + 로켓배송 전용 SLA 판정 Cut-off(자정)까지 주문 → 하루 안에 ALLOCATED 진입
WMS 전국 100+ 풀필먼트 센터(FC) 자체 운영 직매입·직보관·직배송. 피킹 동선 최적화가 핵심 경쟁력
TMS 쿠팡로지스틱스서비스(CLS) 자체 배송망 허브·캠프·기사 수직 통합 → 데이터 일관성↑, 리드타임↓
Last-mile 쿠팡친구 기사 앱, 문앞 배송, 실시간 위치 공유 POD(배송증빙) 사진 자동 촬영, 고객 실시간 추적
Returns 앱 반품 신청 → 같은 기사가 다음 방문 때 수거 정방향·역방향 통합 → 역물류 리드타임 대폭 단축
💡 컬리 샛별배송 — 시간 제약이 모든 설계를 지배
Cut-off = 23:00 / Delivery = 다음날 07:00 이전. 약 8시간 안에 피킹·패킹·간선·라스트마일을 전부 완료해야 함.
→ WMS Wave picking(집중 피킹), TMS 야간 간선 운영, 냉장 Cold-chain 유지가 모두 이 제약에서 파생.
