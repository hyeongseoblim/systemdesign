URL Shortener 시스템 설계

긴 URL을 짧게 줄이고 리다이렉트하는 서비스. 단순해 보이지만 읽기:쓰기 = 100:1의 극단적 읽기 편향, 짧은 코드 생성 전략, 캐시 설계가 면접의 핵심이다. 6단계 프레임워크로 설계한다.

📌 현재 ①~③ 단계까지 완료. ④ High-level 아키텍처 / ⑤ Deep-dive / ⑥ Trade-off 는 다음 세션에서 이어서 작성 예정. 이어서 진행하려면 채팅에서 /design URL Shortener 후 "④부터"라고 요청.

① 요구사항 명확화

URL Shortener는 어떤 요구사항이 붙느냐에 따라 설계가 완전히 달라진다. 면접관이 후보의 깊이를 가장 먼저 보는 단계.

Functional (기능 요구사항)

기능포함 여부비고
긴 URL → 짧은 URL 생성 + 리다이렉트✅ 필수핵심
Custom alias (예: sho.rt/my-event)✅ 포함차별화 포인트
만료 시간(TTL) 설정✅ 선택적 지원기본은 무한
클릭 수(click_count) 카운트✅ 기본만상세 analytics(국가/Referrer)는 제외
내 URL 목록 관리✅ 로그인 사용자익명도 생성은 가능

Non-functional (비기능 요구사항)

항목결정이유
규모중규모 — DAU 1,000만, 일일 신규 10M면접 표준 가정
가용성High (리다이렉트는 99.9%+)죽으면 모든 단축 링크가 깨짐
일관성Eventual 허용 (click_count), 생성은 Strong카운트는 약간 틀려도 됨
지연리다이렉트 p99 < 50ms사용자 체감 직결
지역단일 리전(한국)글로벌은 Trade-off에서 언급
코드 길이7자 Base6262^7 ≈ 3.5조
🎯 면접 포인트 — 같은 긴 URL을 두 번 축약하면?
매번 다른 짧은 URL 반환(비결정적) vs 항상 같은 코드 반환(중복 제거). 후자는 저장 공간 절약되지만 long_url에 인덱스/해시가 필요해 쓰기가 복잡해진다. 본 설계는 bit.ly처럼 비결정적 채택. (중복 제거는 Deep-dive 주제)

② 용량 추정 (Back-of-the-envelope)

핵심 가정

항목값근거
DAU10M중규모
일일 신규 URL10M—
읽기:쓰기 비율100 : 11개 생성 → 100번 클릭 (업계 평균)
보존 기간5년—

QPS 계산

# Write (생성) 일일 10M / 86,400초 ≈ 116 QPS (평균) 피크 ×5 ≈ 580 QPS (피크) → MySQL 단일 인스턴스로 충분 # Read (리다이렉트) 일일 10M × 100 = 1B (10억) 리다이렉트 1B / 86,400초 ≈ 11,574 QPS (평균) 피크 ×5 ≈ 58,000 QPS (피크) → Redis 캐시 필수!

저장량 / 캐시

레코드 1건 ≈ 200B (short_code 7B + long_url ~100B + 메타 + 인덱스) 일일: 10M × 200B = 2 GB/일 1년: 2 GB × 365 ≈ 730 GB 5년: ≈ 3.6 TB → 단일 DB 한계 근처, 향후 샤딩 검토 캐시 (80/20, 최근 7일 핫 URL): 2M 핫/일 × 200B × 7일 ≈ 2.8 GB → Redis 1대로 충분

코드 길이 검증

Base62 7자 = 62^7 ≈ 3.5조 5년 누적 생성 = 10M × 365 × 5 ≈ 18B (180억) 충돌 가능성 = 18B / 3.5조 ≈ 0.5% → 7자로 결정 (6자는 5년 후 위험)
핵심 정리
Write 580 QPS(RDBMS OK) / Read 58K QPS(캐시 필수) / 5년 3.6TB / 캐시 3GB. 이 문제의 첫 번째 난제는 "Read 58K QPS를 어떻게 처리하느냐" — 캐시 hit rate와 DB fallback이 ⑤ Deep-dive의 핵심.

③ API & 데이터 모델

핵심 엔드포인트

메서드경로설명
POST/api/v1/urlsURL 생성 (long_url, custom_alias?, expires_at?)
GET/{short_code}리다이렉트 (302) — 98% 트래픽
GET/api/v1/urls/{short_code}메타데이터 조회 (소유자/admin)
GET/api/v1/users/me/urls?cursor=&limit=내 URL 목록 (Cursor 페이지네이션)
🎯 면접 단골 — 301 vs 302?
301 Moved Permanently: 브라우저가 캐싱 → 다음 클릭부터 서버 미도달 → click_count 측정 불가, 대신 인프라 부담↓.
302 Found: 매번 서버 경유 → 분석 가능, 부담↑.
→ 클릭 분석이 필요하므로 302 채택. 분석이 불필요하면 301이 비용 유리.

데이터 모델 — urls 테이블

CREATE TABLE urls (
  short_code   VARCHAR(7)   NOT NULL,
  long_url     TEXT         NOT NULL,
  user_id      BIGINT       NULL,
  created_at   TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
  expires_at   TIMESTAMP    NULL,
  click_count  BIGINT       NOT NULL DEFAULT 0,

  PRIMARY KEY (short_code),                          -- Clustered Index
  INDEX idx_user_created (user_id, created_at DESC), -- 내 URL 목록용
  INDEX idx_expires (expires_at)                     -- TTL 정리 배치용
) ENGINE=InnoDB CHARSET=utf8mb4;

인덱스 결정

인덱스목적이유
PK = short_code리다이렉트(98% 트래픽)Clustered Index → 행 데이터 즉시 접근, extra lookup 회피
(user_id, created_at DESC)내 URL 목록Leftmost prefix: 필터+정렬 모두 인덱스로
expires_at만료 청소 배치PG는 partial index가 유리
🎯 면접 포인트 — PK를 short_code로? auto-inc ID로?
PK = short_code: 리다이렉트 1회 인덱스 탐색으로 끝(clustered). 단점 → 랜덤 코드라 B+Tree 페이지 단편화(임의 위치 insert).
PK = auto-inc + UNIQUE(short_code): insert 친화적이나 리다이렉트 시 secondary→PK lookup 2회.
→ 읽기:쓰기 100:1이므로 읽기 최적화(short_code PK) 채택.

💰 Free Tier 실배포 MVP

위 설계는 DAU 10M 가정의 면접용 풀스케일. 개인 학습/사용 용도라면 트래픽이 비교가 안 되게 작으므로(Read 1 QPS 미만), 0원으로 충분히 운영 가능하다.

💡 개인 용도 실제 규모
하루 URL 수십 개 생성, 클릭 수백 번 → Read 1 QPS도 안 됨. Oracle Cloud Always Free의 LB 10Mbps 병목에도 절대 안 닿는다. Redis·Kafka·LB 다 빼고 앱 1개 + DB 1개로 시작해도 동작.
