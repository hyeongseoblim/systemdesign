🎓 Concept 모드 system-design-coach Session #0 — 방법론 개관

시스템 디자인 면접 방법론 개관

설계 능력보다 사고 과정과 커뮤니케이션이 평가된다. 6단계 프레임워크를 몸에 익히는 것이 목표.

1. 왜 시스템 디자인 면접인가

시스템 디자인 면접은 "정답이 없는" 열린 문제다. 면접관이 보는 것은 요구사항을 구체화하는 능력, 불확실성 속에서 합리적 결정을 내리는 능력, Trade-off를 명확하게 설명하는 능력이다.

평가 항목잘하는 후보못하는 후보
요구사항 파악Functional/Non-functional 나눠서 명확화바로 설계 시작
용량 추정숫자로 근거 제시 (DAU → QPS → 저장량)추정 생략하거나 근거 없이 숫자
Trade-off"A를 선택하면 B를 잃는다"를 명시"이게 제일 좋습니다" 단정
Deep-dive병목·장애 시나리오 선제적 언급물어볼 때만 반응
커뮤니케이션생각을 소리로 설명하며 진행침묵 후 결과만 제시
💡 핵심 마인드셋
"완벽한 설계"가 아니라 "최선의 절충"을 보여주는 것이 목표. 45분 안에 모든 것을 다 설계할 수 없다는 것을 면접관도 안다.

각 단계별 핵심 질문

단계반드시 물어볼 것흔한 함정
① 요구사항DAU, 읽기/쓰기 비율, SLA(가용성/지연), 데이터 보존 기간, 지역(단일/멀티 리전)비기능 요구사항 생략

③ API/데이터핵심 엔드포인트 3개 내외, 주요 테이블과 인덱스완벽한 스키마 설계에 시간 낭비
④ High-level단일 장애점(SPOF), 데이터 흐름, 확장 포인트컴포넌트 이름만 나열
⑤ Deep-dive가장 어려운 1-2개 집중 (면접관과 합의)모든 것을 얕게 다룸
⑥ Trade-off"왜 이 방식을 선택했나", "대안은 무엇이었나"장점만 설명, 단점 생략

4. 용량 추정 치트시트

📐 기본 수치 암기
시간: 1일 ≈ 86,400초 ≈ 105초  |  1월 ≈ 2.5×106초
DAU → QPS: DAU 1억 → 평균 QPS ≈ 1,200 / 피크 ≈ 5,000–12,000 (피크계수 4–10배)
크기: 텍스트 포스트 ≈ 300B / 사진 ≈ 200KB / 10분 동영상 ≈ 50MB
지연: SSD ≈ 0.1ms / DC 내부 RTT ≈ 0.5ms / 대륙 간 RTT ≈ 150ms
처리량: MySQL ≈ 1,000–10,000 QPS / Redis ≈ 100,000 QPS / Kafka ≈ 수백만 msg/s

추정 예시 — DAU 1천만 Twitter류 서비스

항목계산결과
평균 쓰기 QPS1천만 ÷ 86,400 × 사용자당 3트윗/일≈ 350 QPS
피크 쓰기 QPS350 × 피크계수 5≈ 1,750 QPS
읽기 QPS쓰기의 100배 (읽기 중심 서비스)≈ 35,000 QPS (피크)
일일 저장량350 QPS × 86,400 × 300B≈ 9 GB/일
5년 저장량9 GB × 365 × 5≈ 16 TB
🎯 면접 포인트
추정 과정을 소리로 말하면서 진행. 숫자가 틀려도 OK — 중요한 건 합리적 추론 과정.

5. Trade-off 사고법

축선택지 A선택지 B결정 기준
일관성 vs 가용성Strong Consistency (항상 최신)Eventual Consistency (가용성↑)금융·결제: Strong / SNS 피드: Eventual
정규화 vs 반정규화정규화 (쓰기 효율↑)반정규화 (읽기 효율↑)읽기:쓰기 비율
Pull vs PushPull (소비자가 요청)Push (서버가 밀어넣기)팔로워 수 분포 (팬아웃 비용)
SQL vs NoSQLRDBMS (ACID, 복잡한 쿼리)NoSQL (수평 확장, 유연한 스키마)쿼리 패턴, 정합성 요구 수준
동기 vs 비동기동기 (즉시 응답)비동기 (Queue, 높은 처리량)사용자 응답 필요 여부

6. CAP / PACELC 기초

flowchart TB CAP["CAP 정리\n분산 시스템은 Partition 발생 시\nConsistency 또는 Availability 중\n하나만 보장 가능"] CP["CP 시스템\nConsistency + Partition tolerance\n예: HBase, Zookeeper\n→ Partition 시 쓰기 거부"] AP["AP 시스템\nAvailability + Partition tolerance\n예: Cassandra, DynamoDB\n→ 일시적 불일치 허용"] CA["CA 시스템\n현실에서 불가\nPartition 없는 단일 서버만 가능"] CAP --> CP CAP --> AP CAP -.->|"분산 시스템에서 불가"| CA style CP fill:#dbeafe,stroke:#3b82f6 style AP fill:#dcfce7,stroke:#22c55e style CA fill:#f1f5f9,stroke:#94a3b8,color:#94a3b8
CAP 정리 — 분산 시스템 설계의 핵심 제약. 실무에서는 Partition Tolerance 는 필수.
⚠️ 흔한 오해
"CA를 선택하겠습니다"는 면접에서 감점. 분산 환경에서 P는 포기할 수 없음.
PACELC: Partition이 없을 때도 Latency(지연) vs Consistency 선택이 있음. 더 현실적 프레임워크.
