서비스 경계를 어떻게 나눌지, 분산 환경에서 정합성을 어떻게 보장할지 — MSA·DDD·Saga·Outbox·CQRS 전반을 한눈에 그리기.

1. Modular Monolith vs MSA

flowchart LR subgraph MM["Modular Monolith"] direction TB M1["Order Module"] M2["Inventory Module"] M3["Shipping Module"] M1 --- M2 --- M3 DB1[("단일 DB")] M3 --- DB1 end subgraph MSA_["MSA"] direction TB S1["Order Service"] S2["Inventory Service"] S3["Shipping Service"] DB2[("Order DB")] DB3[("Inventory DB")] DB4[("Shipping DB")] Q["Message Queue\nKafka / SQS"] S1 --> Q --> S2 Q --> S3 S1 --- DB2 S2 --- DB3 S3 --- DB4 end style MM fill:#dbeafe,stroke:#3b82f6 style MSA_ fill:#ede9fe,stroke:#8b5cf6
Modular Monolith vs MSA — 대부분의 팀은 Modular Monolith로 시작해 경계가 안정된 후 분리
관점Modular MonolithMSA
배포 독립성없음 (단일 배포)있음 (서비스별 독립)
데이터 정합성로컬 트랜잭션으로 쉬움분산 트랜잭션 (Saga 필요)
장애 격리부분적좋음 (설계에 따라)
운영 복잡도낮음높음 (관측성·배포·네트워크)
팀 규모소~중 (10명 이하)중~대 (팀당 한 서비스)
초기 속도빠름느림 (인프라 비용)
⚠️ 조기 MSA는 재앙
"MSA 니까 쪼개야 한다"는 금물. 비즈니스 이유 + 팀 규모 + 도메인 성숙도가 준비됐을 때만. 데이터베이스를 공유하면서 서비스만 나눈 것은 Distributed Monolith(분산 모놀리스) — 최악.

서비스 경계 결정 기준 (Conway's Law 의식)

  • 비즈니스 Capability 기반: 함께 변하는 것은 함께 둔다
  • 데이터 소유권: 한 서비스가 한 데이터의 단일 진실 원천(SSOT)
  • 팀 구조와 일치: Team Topologies — Stream-aligned team
  • 채터링(Chattering) 확인: 두 서비스 간 동기 HTTP가 많으면 경계 재고

2. DDD (Domain-Driven Design) 기초

flowchart TB subgraph Strategic["전략 설계 (Strategic Design)"] UL["Ubiquitous Language\n유비쿼터스 언어\n도메인 전문가와 동일 용어"] BC["Bounded Context\n경계 있는 컨텍스트\n모델의 경계"] CM["Context Map\n컨텍스트 간 관계\nACL / Partnership / Conformist"] UL --> BC --> CM end subgraph Tactical["전술 설계 (Tactical Design)"] AG["Aggregate\n애그리거트\n일관성 경계 + Root"] DE["Domain Event\n도메인 이벤트\n과거형 동사"] VO["Value Object\n값 객체\n불변, 동등성"] DS["Domain Service\n도메인 서비스\nEntity에 맞지 않는 로직"] AG --> DE end Strategic --> Tactical style Strategic fill:#dbeafe,stroke:#3b82f6 style Tactical fill:#ede9fe,stroke:#8b5cf6
DDD 전략 설계 → 전술 설계 순서. 전략 없이 전술만 하면 Aggregate가 엉뚱하게 설계됨.
개념정의핵심 규칙
Bounded Context모델이 유효한 경계. 같은 단어가 컨텍스트마다 다른 의미"상품" — 카탈로그 vs 재고 vs 정산 다름
Aggregate일관성 경계. Aggregate Root 를 통해서만 접근한 트랜잭션 = 한 Aggregate 변경
Domain Event도메인에서 발생한 사실. 과거형 명사OrderPlaced, InventoryReserved
ACL (Anti-Corruption Layer)다른 BC의 모델을 내 모델로 변환하는 레이어외부 모델이 내 도메인을 오염시키지 않도록

4. Saga 패턴 — 분산 트랜잭션 대안

sequenceDiagram participant Orch as Saga Orchestrator participant Order as Order Service participant Inv as Inventory Service participant Pay as Payment Service Note over Orch,Pay: 정상 흐름 Order->>Orch: OrderPlaced Orch->>Inv: ReserveInventory Inv-->>Orch: InventoryReserved Orch->>Pay: ProcessPayment Pay-->>Orch: PaymentProcessed Orch->>Order: ConfirmOrder Note over Orch,Pay: 결제 실패 시 보상 트랜잭션 Orch->>Pay: ProcessPayment Pay-->>Orch: PaymentFailed Orch->>Inv: ReleaseInventory Inv-->>Orch: InventoryReleased Orch->>Order: CancelOrder
Saga Orchestration — 중앙 오케스트레이터가 흐름 제어 + 실패 시 보상(Compensation)
🎯 면접 포인트 — 2PC vs Saga
2PC(Two-Phase Commit): Blocking, 코디네이터 SPOF, 가용성↓ → 현대 분산 시스템에서 거의 사용 안 함.
Saga: 각 스텝이 로컬 트랜잭션 + 실패 시 보상(Compensating transaction). Eventual Consistency 수용.
보상 트랜잭션 설계 누락은 면접에서 치명적. 🔥 Deep-dive

5. Outbox 패턴 & Idempotency

sequenceDiagram participant Svc as Service participant DB as Database participant Relay as Outbox Relay (CDC) participant MQ as Message Queue Note over Svc,MQ: DB 커밋 + 이벤트 발행 원자화 Svc->>DB: BEGIN TRANSACTION Svc->>DB: INSERT INTO orders (...) Svc->>DB: INSERT INTO outbox (event_type, payload, status='PENDING') Svc->>DB: COMMIT Note over DB,MQ: 비동기 릴레이 Relay->>DB: SELECT * FROM outbox WHERE status='PENDING' DB-->>Relay: 이벤트 목록 Relay->>MQ: Publish(event) MQ-->>Relay: ACK Relay->>DB: UPDATE outbox SET status='SENT'
Outbox 패턴 — DB 트랜잭션과 메시지 발행을 원자적으로. Relay는 Debezium(CDC)이 일반적.

Idempotency (멱등성) 보장

  • 멱등성 키(Idempotency-Key): 클라이언트가 요청마다 고유 키 포함. 서버는 처리 결과를 저장해 중복 요청 시 같은 결과 반환.
  • 저장 위치: Redis (TTL 24h) 또는 DB (unique constraint).
  • Kafka 소비자 멱등성: At-least-once 전달 특성 → 동일 이벤트 두 번 처리 방어 로직 필수.
⚠️ 핵심 함정 — DB 커밋 후 이벤트 발행 금지
orderRepository.save(order) // 커밋
kafkaTemplate.send("order-created", event) // 여기서 장애나면?
→ 이벤트 유실. Outbox 패턴으로 DB 트랜잭션에 이벤트를 함께 포함시켜야 함.

6. CQRS (Command Query Responsibility Segregation) 개관

flowchart LR Client(["Client"]) subgraph Write["쓰기 모델 (Command)"] CMD["Command Handler"] WDB[("Write DB\nRDBMS")] DE["Domain Events"] CMD --> WDB CMD --> DE end subgraph Read["읽기 모델 (Query)"] QH["Query Handler"] RDB[("Read DB\n역정규화 뷰\nElasticsearch/Redis")] QH --> RDB end Client -->|"쓰기 요청"| CMD Client -->|"읽기 요청"| QH DE -->|"이벤트로 동기화"| RDB style Write fill:#fce7f3,stroke:#ec4899 style Read fill:#dcfce7,stroke:#22c55e
CQRS — 쓰기 모델과 읽기 모델 분리. 읽기 모델은 쿼리에 최적화된 별도 구조.
💡 CQRS 적용 기준
적용해야 할 때: 읽기/쓰기 확장 요구가 극단적으로 다름, 복잡한 집계 뷰가 필요, Event Sourcing 과 결합.
적용하면 안 될 때: 단순 CRUD, 팀이 작음, 최종 일관성(Eventual Consistency)이 비즈니스적으로 허용 안 됨.
CQRS 과잉 적용은 복잡도 폭발로 이어짐. 필요성을 먼저 증명하라.
