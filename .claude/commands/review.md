# 코드/구현 리뷰 모드

이 세션은 jobStudy CLAUDE.md 의 "코드/구현 리뷰 모드" 규칙을 따른다. 단순 스타일 지적이 아니라 **시니어 관점의 구조적 피드백**을 목표로 한다.

## 진행 규칙

### 1. 코치 선택
- Spring/Kotlin/Java/Go 코드 → `backend-dev-coach` (메인)
- 아키텍처 설계 이슈 동반 → `backend-architecture-coach` 병행
- SQL/쿼리/인덱스 → `database-coach` 병행
- K8s 매니페스트/Terraform/CI 스크립트 → `infra-coach`
- 자료구조·알고리즘·성능 → `cs-fundamentals-coach`

### 2. 입력 형태
- 파일 경로가 주어지면 Read 로 읽어서 리뷰
- PR URL 이 주어지면 변경 사항 파악
- 코드 블록이 직접 붙여넣어져 있으면 그대로 리뷰
- 맥락이 부족하면 **먼저 질문** (용도, 호출 빈도, 동시성 환경, SLA 등)

### 3. 리뷰 체크리스트 (반드시 전부 순회)

| 항목 | 체크 포인트 |
|---|---|
| **정확성** | 요구사항 충족? 엣지 케이스(null, empty, 0, 음수, 경계값)? off-by-one? |
| **동시성 안전성** | 공유 가변 상태? 적절한 락/원자 연산? Race condition? Deadlock 가능성? |
| **트랜잭션 경계** | `@Transactional` 위치 적절? 자기호출·private·async 함정? 외부 호출이 트랜잭션 안에? |
| **에러 처리** | 예외 swallow 금지. 재시도 전략(백오프)? Timeout? Fallback? Checked vs Runtime? |
| **테스트 가능성** | 순수 함수 분리? 의존성 주입? Testcontainers 가능한가? |
| **관측성** | 구조화 로그(JSON + traceId/spanId)? 메트릭(RED/USE)? 분산 추적? PII 마스킹? |
| **성능** | N+1 쿼리? 인덱스 활용? 불필요한 연산? 메모리 할당? |
| **보안** | 입력 검증? SQL 인젝션·XSS? 시크릿 하드코딩? 권한 체크 위치? |
| **가독성** | 함수 책임 단일? 명명? 매직 넘버? 주석은 왜(Why) 중심? |

### 4. 피드백 형식

각 지적 사항은 다음 구조로:

```
### [심각도] 항목명 (파일:라인)
**문제**: 무엇이 왜 문제인가
**영향**: 어떤 상황에서 어떻게 망가지는가 (구체 시나리오)
**제안**:
```diff
- 기존 코드
+ 수정 코드
```
**근거**: 왜 이 방향이 나은가 (Trade-off 있으면 명시)
```

심각도:
- 🔴 **Blocker**: 프로덕션 배포 금지 수준 (버그/보안/데이터 손실)
- 🟠 **Major**: 배포는 가능하나 반드시 후속 작업 필요
- 🟡 **Minor**: 개선 권장
- 🔵 **Nit**: 스타일·취향 (선택적)

### 5. 긍정적 피드백도 함께
잘 된 부분도 짧게 언급 — 무엇을 유지하면 좋을지 알려준다.

### 6. 마지막에 요약
- **Blocker/Major 개수**
- **다음 액션 우선순위** 3-5개
- 가능하면 **후속 학습 주제** 제안 (Concept/Design 모드로 연결)
